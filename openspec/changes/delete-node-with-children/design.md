## Context

当用户删除一个有子节点的节点时，当前系统会将子节点的 `top` 字段（原父节点 ID）保留，导致子节点找不到父节点，在 UI 上显示异常。

用户选择的产品愿景是：**知识整理笔记软件，树状结构，节点即笔记**。层级关系代表知识的组织方式，需要慎重处理。

## Goals / Non-Goals

**Goals:**
- 空节点直接删除，无需额外操作
- 有子节点的节点删除时，提供明确的用户选择
- 子节点提升时，正确更新数据库中的 `top` 字段

**Non-Goals:**
- 不实现级联删除（用户主动选择时才删除子树）
- 不自动将子节点变成根节点（必须挂到有效父节点下）

## Decision: 用户选择对话框

当删除有子节点的节点时，弹出确认对话框：

```
┌─────────────────────────────────────┐
│          确认删除节点               │
├─────────────────────────────────────┤
│  节点 "A" 包含 3 个子节点。          │
│  请选择删除方式：                   │
│                                     │
│  ○ 删除整个子树                     │
│    删除 "A" 及其所有子节点          │
│                                     │
│  ○ 仅删除 "A"                       │
│    子节点将提升到上一级             │
│                                     │
│           [取消]  [确认]            │
└─────────────────────────────────────┘
```

**选项解释：**
- **删除整个子树**：不可逆操作，删除选中节点及其所有后代节点
- **仅删除父节点**：删除选中节点，子节点自动挂到祖父节点下，保持层级

## 数据模型变更

### 子节点提升逻辑

```
删除节点 A（A.parent = P，A.children = [B, C]）
   ↓
B.top = A.top（即 P）→ B 现在是 P 的子节点
C.top = A.top（即 P）→ C 现在是 P 的子节点
A 被删除
```

### 数据库操作

1. 查询 A 的所有子节点（包括递归子节点）
2. 如果用户选择"仅删除父节点"：
   - 将所有子节点的 `top` 字段更新为 `A.top`
3. 删除节点 A（包括数据库记录和文件）

## Risks / Trade-offs

- **风险**：子节点提升后，如果用户在删除前没有感知到层级变化，可能造成知识结构不符合预期
- **缓解**：对话框清晰说明操作效果
- **权衡**：对话框增加了一步操作，但提供了明确的控制权
