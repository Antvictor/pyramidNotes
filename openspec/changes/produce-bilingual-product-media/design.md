## Context

### 当前自动化现状

仓库中已经有两类媒体生成脚本：

1. `scripts/capture-product-media.mjs`
   - 负责生成静态截图
   - 当前输出尺寸固定为 `1440x980`

2. `scripts/capture-product-workflow.mjs`
   - 负责生成 workflow GIF 与 `webm`
   - 当前录制尺寸固定为 `1280x860`
   - 当前使用 `canvas.captureStream(6)`，约等于 `6fps`
   - 当前每一步停留时间大致在 `700ms - 1400ms`

### 当前问题

1. **官网展示位升级后，静态图分辨率不够**
   - `1440x980` 更像普通展示图，而不是 Hero / 大舞台级素材
   - 文字内容在官网大图位里可读性不稳定

2. **workflow 视频编码策略不适合桌面产品 UI**
   - 视频体积极小，说明压缩过强
   - 低帧率 + 快节奏 + 强压缩叠加后，导致界面文字不清晰

3. **缺少正式规范文档**
   - 当前参数散落在脚本里
   - 后续重录时缺少“应该输出到什么标准”的统一依据

## Goals / Non-Goals

**Goals**

- 定义官网截图与 workflow 视频的采集规范
- 升级现有脚本，使其可自动生成更适合官网的高清中英文素材
- 重新产出 `website/assets/product-media/{zh,en}` 中的关键媒体
- 保持自动化流程可重复执行

**Non-Goals**

- 不在本次范围内生成全平台营销尺寸成品
- 不引入伪造 UI 的 AI 图片生成流程
- 不修改 website 页面结构本身
- 不读取真实用户笔记或真实工作数据

## Decisions

### Decision 1: 区分“官网截图标准”和“营销平台扩展标准”

**选择**

- 本次规范以官网展示位为主
- 文档中补充“后续扩展到营销平台”的建议，但不要求这次全部生成
- 规范文档明确标注为 `website-first` 基准规范

**理由**

- 当前最紧急的问题来自 website
- 官网使用位最依赖高清可读性
- 一次性把所有营销渠道尺寸也纳入，会显著放大本次 change 范围

### Decision 2: 静态截图需要更高输出尺寸

**选择**

- 将官网主截图目标提升到高于当前 `1440x980` 的输出级别
- 推荐标准：输出宽度至少 `2200px`，优先考虑约 `2560px` 宽

**理由**

- 当前官网的大舞台展示位已经明显超过现有截图的舒适显示范围
- 更大的源图能更好支撑高分屏和产品细节展示

**实现要点**

- 增大 CDP 视口尺寸
- 保证截图来自更大的应用窗口，而不是对小窗口截图进行后期放大
- 对桌面 UI 图优先保真，不用高压缩格式替代 PNG
- 最终验证必须以 website 当前 Hero / 功能演示展示位的直接可读性为准

### Decision 3: workflow `webm` 优先重录而不是仅替换编码参数

**选择**

- 同时调整“录制节奏”和“编码参数”

**理由**

- 当前问题不仅是模糊，还有“太快”
- 仅提高码率，不能解决关键步骤停留过短的问题
- 仅延长停留时间，也无法解决过低帧率和压缩过狠的问题

**实现要点**

- 提升 workflow 录制尺寸
- 提升 capture stream 帧率，目标至少高于当前 `6fps`，推荐提升到 `10fps - 12fps`
- 增加步骤间停留时长，单步停留目标提升到约 `1200ms - 2200ms`
- 输出更适合官网展示的 `webm`
- 总体时长允许明显长于当前版本，只要关键步骤可读

### Decision 4: workflow 用 `webm` 作为官网主视频，GIF 作为兼容副产物

**选择**

- `webm` 继续作为官网嵌入主格式
- GIF 保留为辅助输出，不作为官网主显示媒介

**理由**

- `webm` 在相同质量下更适合控制体积
- GIF 对 UI 文字展示通常更差，只适合兼容或外部分发

### Decision 5: 建立明确的官网素材质量规范

**选择**

- 新增一份可读的素材规范文档

**内容至少包括**

- 静态截图目标尺寸
- 应用窗口尺寸
- 取景规则
- 必拍场景列表
- workflow 步骤与建议停留时长
- 中英文语言一致性要求
- 隐私与真实性检查清单

### Decision 6: 重新生成 website 当前使用的关键媒体即可

**选择**

- 优先重做当前 website 已引用的关键媒体
- 默认覆盖 `website/assets/product-media/{zh,en}` 下当前同名素材

**范围**

- `tree overview`
- `editor open note`
- `search fulltext`
- `node operations`
- `workflow webm`

**理由**

- 这些正是当前官网实际展示位使用的素材
- 不必先重做所有边缘场景截图

## Media Specification

### Static Screenshot Spec

- 输出用途：website Hero、大舞台功能演示、独立截图展示
- 推荐输出宽度：约 `2560px`
- 输出格式：PNG
- 原则：
  - 用更大的应用窗口直接截图
  - 保证文字可读优先于“铺满”
  - UI 不做伪造或手工改字
  - 中英文场景结构保持对应
  - 最终在 website 当前展示位中无需“新标签打开再放大”也应可读

### Workflow Video Spec

- 输出用途：website workflow section
- 输出格式：WebM
- 推荐策略：
  - 更高画布尺寸
  - 更高帧率，推荐 `10fps - 12fps`
  - 更长停留时间，推荐单步 `1200ms - 2200ms`
  - 总时长可接受比当前更长，只要能看清

- 推荐步骤节奏：
  - `tree`: 停留更久，先让用户识别结构
  - `note`: 停留更久，确保编辑区可读
  - `note-scroll`: 滚动速度慢于当前
  - `back-tree`: 给短暂停顿
  - `search-fulltext`: 结果停留足够长

## Verification

1. 新生成截图在官网大图位中能直接看清主要 UI 文字
2. 新生成 `webm` 在官网中播放时，关键步骤不会一闪而过
3. 中英文素材路径与 website 当前引用保持兼容
4. 脚本支持重复执行，不依赖手工逐张处理
5. 生成前后没有使用真实用户笔记数据
6. website 当前引用无需修改或只需最小修改即可吃到新素材
